Today's AI agents are task-oriented.

They wait.

They forget.

Most systems operate like this:

```text

```

* Knowledge becomes fragmented
* Learning is temporary
* Context is lost
* Intelligence does not compound

Businesses do not need more agents.

## What we're building

A **hosted workspace** where you hire AI employees the same way you'd hire a real person — they read your company, introduce themselves, and start working. Your employees live in a Discord-like interface, post proactively, execute real tool actions, and remember everything. You pay for tokens at cost.

**Reference product: [duet.so](https://duet.so)**
Duet built an always-on AI coworker for small businesses with a beautiful hiring UX and token pass-through billing. ## Hiring flow (onboarding)

Four steps, five minutes. At the end the user has a hired employee who already knows their business.

**Step 1 — Company**
- Company name + website URL
- AI reads the website in real-time while the user watches ("Reading nareshit.com...")
- Extracts: company description, products/services, tone, industry, team size
- User confirms or corrects — takes 30 seconds

**Step 2 — Sources**
- Optionally connect GitHub, Gmail, Slack, or HubSpot (OAuth)
- Employee will use these as live data sources from day one
- Skip to connect later

**Step 3 — Your new employee**
- Pick role: CEO Assistant / Marketing / Sales / Support / Finance / HR / Custom
- Name them (or accept a generated name like "Prasadd")
- Voice: Casual / Professional / Direct
- Agency: Proactive (acts without being asked) / Reactive (responds on demand)
- Mode: Commander (asks before doing) / Run Ahead (just does it)

**Step 4 — Hire**
- Preview the EMPLOYEE ID card: name, org, role, hire date, voice, agency, barcode
- Click HIRE
- Confetti. Card stamped HIRED. Employee drops into the workspace and posts their first message.

The employee's first message is always proactive: they've already read the company, identified the top 2–3 priorities, and ask one focused question before starting.

## Technical architecture (cloud layer)

Built on top of the existing monorepo. New packages added:

```
employeeos/
├── apps/
│   ├── terminal/          (existing — open source CLI)
│   ├── gateway/           (existing — local HTTP server)
│   ├── web/               (existing — local React dashboard)
│   └── cloud/             (NEW — cloud web app, extends apps/web)
│       ├── src/
│       │   ├── Workspace.tsx     Discord-like main view
│       │   ├── Onboarding.tsx    4-step hiring flow
│       │   ├── HireCard.tsx      Employee ID card + barcode
│       │   ├── EmployeeChat.tsx  Per-employee channel view
│       │   ├── PlanCard.tsx      Inline plan approve/reject
│       │   └── Billing.tsx       Usage dashboard
│       └── vite.config.ts
│
└── packages/
    ├── (all existing packages — unchanged)
    ├── tenant/            (NEW) Multi-tenant isolation
    │   ├── createTenant() — provisions DB, worker slot, credential vault
    │   ├── getTenantDB()  — returns isolated DatabaseService per org
    │   └── schema/        — same Drizzle schema, namespaced per tenant
    ├── billing/           (NEW) Stripe integration
    │   ├── createMeter()  — Stripe metering event for each token batch
    │   ├── getUsage()     — real-time usage for dashboard
    │   └── syncSubscription() — webhook handler for Stripe events
    ├── vault/             (NEW) Encrypted credential storage
    │   ├── setSecret()    — AES-256 encrypt + store per tenant
    │   └── getSecret()    — decrypt on demand for tool calls
    └── worker/            (NEW) Cloud brain loop runner
        ├── scheduleOrg()  — registers org for hourly ticks via BullMQ
        ├── runTick()      — same as brain.runHourlyTick(), posts results to WS channel
        └── streamToChannel() — emits employee messages to connected WS clients
```

**Infrastructure stack (v0.1):**
- Deploy: Railway (monorepo-friendly, managed Postgres, background workers, WebSocket support)
- Auth: Clerk (org management, Google OAuth, multi-tenant built-in)
- Database: Postgres per tenant (RLS) OR SQLite per tenant stored in R2/S3
- Queue: BullMQ (Redis) — one recurring job per active org
- Real-time: WebSocket (existing gateway extended) — employee messages stream to browser
- Storage: Cloudflare R2 — documents, screenshots, CSV imports

**Why Railway for v0.1:** Handles monorepo deploy out of the box, managed Redis for BullMQ, easy horizontal scaling when needed. Migrate to Fly.io or AWS ECS when volume demands it.

## What the employee does on day 1 (example: NareshIT)

1. Onboarding completes — "Priya" hired as Marketing Manager, read nareshit.com
2. Priya posts in #marketing: *"Hey — I've scanned the NareshIT site. You have 40+ courses across Python, Java, DevOps, Data Science. I see no YouTube content in the last 30 days and the blog hasn't been updated since March. Biggest opportunity I see: YouTube Shorts series on course comparisons. Want me to draft the first 5 scripts?"*
3. User replies: "yes go ahead"
4. Priya makes 5 tool calls (reads site more deeply, checks similar channels, generates scripts), posts structured 5-episode shot-by-shot table
5. User approves — Priya creates a plan: `{ tool: "github", operation: "create_issue", input: { title: "YouTube Shorts — Batch 1 scripts ready" } }`
6. Plan card appears inline with [Approve] [Reject] buttons
7. User approves — GitHub issue created, Priya posts confirmation: *"Issue #47 opened. Scripts are in the issue body. Next: want me to schedule these or draft the captions too?"*

This entire flow uses the existing executor, planner, observer, and GitHub integration — zero new AI logic needed.

## Key decisions before building

| Decision | Options | Recommendation |
|---|---|---|
| DB per tenant | SQLite in R2 vs Postgres RLS | Postgres RLS — easier queries, Railway has managed Postgres |
| Auth | Clerk vs Auth.js | Clerk — org management built-in, fastest to ship |
| Brain loop | Always-on vs scheduled | Scheduled (hourly BullMQ job) — free tier orgs paused when inactive |
| Token markup | 0% pass-through vs 15% margin | 15% — needed to cover infra costs; be transparent about it |
| Avatar | 3D characters vs illustrations vs DiceBear | DiceBear for v0.1, upgrade to custom illustrations in v2 |
| Infra | Railway vs Fly.io vs AWS | Railway for v0.1 — zero config monorepo deploy, migrate later |
| Real-time | WebSocket vs SSE | WebSocket — already in gateway, reuse it |
